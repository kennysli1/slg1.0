# 数据存储结构

> 讲清楚 `data/game.json` 里到底存了什么。改数据前先看这篇，才知道动哪张表。
> 代码出处：存储层 `packages/server/src/infra/store.ts`，各表的读写在对应的 `modules/*.ts`。

## 一、总体形态

持久化用**单个 JSON 文件**（默认 `data/game.json`），内部是「类 NoSQL」的两层结构：

```
{
  "<集合名>": {
    "<文档主键>": { ...文档字段 },
    ...
  },
  ...
}
```

- **集合（collection）** ≈ 数据库的表。每个领域模块独占若干集合，别的模块不碰（架构铁律：状态归属唯一）。
- **文档（document）** ≈ 表里的一行，是一个 JSON 对象。
- 存储接口只有 4 个操作：`get / set / delete / all`，外加运维用的 `clear(collection)`（清空整张表）。**没有 SQL、没有 join**；跨表关联靠在文档里存对方的主键（如 player 存 `villageId`）。

数据先在内存里操作（`Map<集合, Map<主键, 文档>>`），变更后防抖 1s 落盘，写盘用「先写 `.tmp` 再 rename」保证原子。进程收到 SIGINT/SIGTERM 会先 flush 再退出（见 `main.ts`）。

## 二、两类数据

刷档时的关键区分（见 `app.ts` 里的 `PROGRESS_COLLECTIONS` / `ACCOUNT_COLLECTIONS`）：

| 类别 | 集合 | 说明 |
|------|------|------|
| **账号**（长期保留） | `player` `player_byname` `player_byvillage` `player_seq` | 登录凭据、种族、地图位置、玩家↔村庄映射 |
| **游戏进度**（一局的状态） | `economy` `building` `military` `movement` `movement_seq` `pve` `world_meta` `world_tile` | 资源、建筑、军队、在途部队、野怪、地图 |

> ⚠️ 新增有状态的模块时，务必判断它的数据属于哪一类，并把集合名登记到 `app.ts` 的对应白名单，否则刷档会漏清或误清。

## 三、各集合明细

主键列写的是「用什么当 key」。字段类型用 TS 记法。

### 账号类

#### `player` — 玩家账号（owner: `modules/player.ts`）
主键：玩家 id（`p-<序号>`，如 `p-1`）

| 字段 | 类型 | 含义 |
|------|------|------|
| `id` | string | 玩家 id，`p-<序号>` |
| `name` | string | 用户名，唯一，≤16 字 |
| `pwd` | string | 密码哈希 `salt:hash`（hex），scrypt 加盐。**绝不存明文，也绝不下发前端** |
| `tribe` | string | 种族：`romans` / `gauls` / `teutons` |
| `villageId` | string | 拥有的村庄 id（`v-<玩家id>`，如 `v-p-1`） |
| `x` `y` | number | 村庄在地图上的坐标 |
| `createdAt` | number | 注册时间戳（ms） |

#### `player_byname` — 用户名→id 索引
主键：用户名（trim 后）；值：玩家 id 字符串。用于登录查号、注册查重。

#### `player_byvillage` — 村庄→玩家 id 索引
主键：villageId；值：玩家 id 字符串。用于 PvP 时按被攻击村反查主人。

#### `player_seq` — 玩家序号计数器
主键固定 `n`；值：number（已发出的最大玩家序号）。注册时自增，保证 id 不重复。

### 游戏进度类

> 下面 economy / building / military 三张表都**以 `villageId` 为主键**（一村一文档），字段里也冗余存一份 `villageId`。

#### `economy` — 村庄经济（owner: `modules/economy.ts`）
| 字段 | 类型 | 含义 |
|------|------|------|
| `villageId` | string | 村庄 id |
| `resources` | `{wood,clay,iron,crop: number}` | 当前 4 资源存量 |
| `lastTick` | number | 上次结算时间戳（惰性结算：读的时候才按流逝时间补产出） |
| `baseRate` | `{wood,clay,iron,crop: number}` | 基础产率（/小时） |
| `rateModifiers` | `{source, mult}[]` | 产率加成层（分层叠加，见架构铁律4） |
| `cropUpkeep` | `Record<string, number>` | 各消耗源的 crop 耗用（人口/军队） |
| `capacity` | `{wood,clay,iron,crop: number}` | 仓库/粮仓容量上限 |

#### `building` — 建筑与资源田（owner: `modules/building.ts`）
| 字段 | 类型 | 含义 |
|------|------|------|
| `villageId` | string | 村庄 id |
| `buildings` | `Record<string, number>` | 中心建筑 code→等级 |
| `fields` | `{type, level}[]` | 18 块资源田，各自类型与等级 |
| `queue` | `{target, fieldIndex?, toLevel, startAt, finishAt, taskId} \| null` | 当前建造队列（一次一项）。`taskId` 关联 Scheduler 定时任务，用于重启恢复 |

#### `military` — 军队与训练（owner: `modules/military.ts`）
| 字段 | 类型 | 含义 |
|------|------|------|
| `villageId` | string | 村庄 id |
| `tribe` | string | 种族（决定可训练兵种） |
| `troops` | `Record<string, number>` | 兵种 code→数量 |
| `smithyLevel` | `Record<string, number>` | 铁匠铺各兵种升级等级 |
| `training` | `{unit, remaining, nextDoneAt, taskId} \| null` | 当前训练订单（逐个产出）。`taskId` 用于重启恢复 |

#### `movement` — 在途部队（owner: `modules/movement.ts`）
主键：行军 id（`m-<序号>`）
| 字段 | 类型 | 含义 |
|------|------|------|
| `id` | string | 行军 id |
| `type` | `'raid' \| 'attack' \| 'return'` | 掠夺 PvE / 进攻玩家 / 返程 |
| `fromVillage` | string | 出发村 id |
| `fromXY` `toXY` | `{x,y}` | 起点/终点坐标 |
| `targetId?` | string | PvE 目标 id（raid 时） |
| `targetVillage?` | string | 被攻击村 id（attack 时） |
| `troops` | `Record<string, number>` | 随行兵力 |
| `loot?` | `Record<string, number>` | 战利品（return 时携带） |
| `departAt` `arriveAt` | number | 出发/到达时间戳 |

#### `movement_seq` — 行军序号计数器
主键固定 `n`；值：number。同 `player_seq`。

#### `pve` — 野怪目标（owner: `modules/pve.ts`）
主键：目标 id（来自 `config/pve_spawns.csv` 的 id）
| 字段 | 类型 | 含义 |
|------|------|------|
| `id` | string | 目标 id |
| `type` | string | 目标模板类型（对应 `pve_targets.csv`） |
| `x` `y` | number | 地图坐标 |
| `defender` | `Snapshot`（`Record<兵种, {...}>`） | 当前守军（被打减员，重生恢复满） |
| `loot` | `Record<string, number>` | 可掠夺资源 |
| `cleared` | boolean | 是否已被清空（待重生） |

#### `world_meta` — 地图元信息（owner: `modules/world.ts`）
主键固定 `meta`；字段：`{ size: number }`（地图范围 `[-size, size]` 方形）。

#### `world_tile` — 地图地块
主键：`"x,y"`（坐标字符串）
| 字段 | 类型 | 含义 |
|------|------|------|
| `x` `y` | number | 坐标 |
| `kind` | `'empty' \| 'village' \| 'pve'` | 地块类型 |
| `refId?` | string | 村庄/目标 id（empty 时无） |
| `name?` | string | 展示名 |
| `icon?` | string | 图标基名（pve 用；渲染拼 `/art/<基名>.png`） |

> 只有 `village`/`pve` 地块会被真正写盘；`empty` 是查询时的默认返回，不占存储。

## 四、主键与 ID 规则

- **玩家 id**：`p-<序号>`，序号由 `player_seq` 自增。
- **村庄 id**：`v-<玩家id>`，如玩家 `p-3` 的村庄是 `v-p-3`。一一对应。
- **行军 id**：`m-<序号>`，由 `movement_seq` 自增。
- **PvE 目标 id**：直接用 `config/pve_spawns.csv` 里配的 id。
- **地块 key**：`"x,y"` 坐标串。

> 注意区分：**config 目录表**（fields/buildings/units/pve_targets）的主键是数字 `id`，跨表引用填数字（见 `config/README.md`）；这里说的是**存档运行时**的实体 id，两者不是一回事。

```bash
node -e "const d=require('./data/game.json'); console.log(Object.values(d.player)[0])"
```
