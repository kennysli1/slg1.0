# 数据库操作手册

> 服务器上对存档的日常运维：查看、备份、手改、刷档、删档、排查。
> 存档结构见 [01_数据存储结构.md](./01_数据存储结构.md)。

## 0. 存档在哪

- 默认路径 `data/game.json`（相对进程 cwd）。
- 生产用环境变量 `DATA_PATH` 指定（pm2 配置见 `ecosystem.config.cjs`，默认 `./data/game.json`）。
- 整个 `data/` 目录被 git 忽略——**存档和备份都不会进版本库**，服务器上是唯一副本，操作前务必先备份。

## 1. 查看存档

```bash
# 各集合文档数概览
node -e "const d=require('./data/game.json'); for(const k of Object.keys(d)) console.log(k.padEnd(18), Object.keys(d[k]).length)"

# 列出所有玩家（名字/种族/坐标；pwd 是哈希）
node -e "const d=require('./data/game.json'); Object.values(d.player).forEach(p=>console.log(p.id, p.name, p.tribe, p.x+','+p.y))"

# 看某村资源
node -e "const d=require('./data/game.json'); console.log(d.economy['v-p-1']?.resources)"
```

> JSON 是单行压缩存的，直接 `cat` 不好读。可以 `node -e "console.log(JSON.stringify(require('./data/game.json'),null,2))" > /tmp/pretty.json` 生成带缩进的版本来看。

## 2. 备份 / 恢复

**刷档命令会自动备份**（见第 4 节），但手动改动前请自己也备一份：

```bash
# 备份（带时间戳）
cp data/game.json "data/game.$(date +%Y%m%d-%H%M%S).json"

# 恢复：停服 → 覆盖 → 起服
pm2 stop kow
cp data/game.20260701-120000.json data/game.json
pm2 start kow
```

> 自动备份统一放在 `data/backups/`，文件名形如 `game.2026-07-01T03-01-25-435Z.json`。定期清理老备份即可。

## 3. 手改存档（谨慎）

**必须先停服**，否则内存里的旧数据会在下次落盘时覆盖你的手改。

```bash
pm2 stop kow                    # 1. 停服（触发一次 flush）
cp data/game.json data/game.bak.json # 2. 备份
# 3. 用编辑器改 data/game.json —— 保持 JSON 合法（逗号/引号/括号）
pm2 start kow                   # 4. 起服，改动生效
```

常见手改场景：给某玩家发资源、改坐标、封号（删 `player`/`player_byname` 里对应文档）。改完起服前可以先用第 1 节的命令验证 JSON 能被 `require` 解析（不报错即合法）。

> 能用配置或命令解决就别手改。手改容易破坏跨表一致性（比如删了 `player` 却漏删 `player_byvillage`）。

## 4. 刷档（重置游戏）

三个命令，**按这次改动的性质选**。三者执行前都会自动备份到 `data/backups/`，且是**独立一次性进程**（不启动游戏、不激活定时任务），跑完即退。

| 命令 | 保留账号 | 保留地图位置 | 清掉什么 | 典型场景 |
|------|:--:|:--:|------|------|
| `npm run reset:season` | ✓ | ✓ | 资源/建筑/军队/在途/野怪/地图，全部归零重建 | 只调了平衡数值、开新赛季，让老玩家原地重来 |
| `npm run reset:respawn` | ✓ | ✗ | 同上，且**重新分配每个账号的地图坐标** | 改了地图尺寸/布局（`game_constants.csv` 的 mapSize、开局模板） |
| `npm run wipe:all` | ✗ | — | **一切**，包括账号 | 大改数据结构、测试完要干净重来 |

**保留账号是什么意思**：用户名、密码、种族都不动，玩家仍能用原密码登录，登录后在（新的或原来的）地图位置上有一座全新的初始村庄。

**怎么选**：
- 只动了 `config/*.csv` 的数值（兵种攻防、成本、产率…）→ `reset:season`。
- 动了地图尺寸或开局布局（村庄要重新摆）→ `reset:respawn`。
- 动了存档字段结构、或就是想彻底清空 → `wipe:all`。

**生产上的完整流程**（务必先停服，避免旧内存覆盖刷档结果）：

```bash
pm2 stop kow
DATA_PATH=./data/game.json npm run reset:season   # 换成你要的模式
pm2 start kow
```

> 本地开发直接 `npm run reset:season` 即可（默认 `data/game.json`）。
> 命令实现见 `packages/server/src/admin.ts`；刷档逻辑在 `app.ts` 的 `resetWorld()`。

## 5. 删档 = wipe:all

不要手动 `rm data/game.json`——那样下次起服虽然也是全新，但你丢了自动备份这层保险。用：

```bash
pm2 stop kow && npm run wipe:all && pm2 start kow
```

`wipe:all` 会先备份再清空，效果等同删档但可回滚。

## 6. 排查数据问题

| 症状 | 查什么 |
|------|--------|
| 玩家登录不上 | `player_byname` 里有没有该用户名 → 指向的 id 在 `player` 里在不在 |
| PvP 攻击提示找不到主人 | `player_byvillage` 里该 villageId 的映射是否还在 |
| 重启后在途任务没了 | `main.ts` 是否走了 `resume()`（有存档才 resume）；`building/military/movement` 文档里 `queue/training/movement` 的 `taskId` 是否还在 |
| 刷档后某模块数据没清/被误清 | 该集合名是否正确登记在 `app.ts` 的 `PROGRESS_COLLECTIONS`（进度）或 `ACCOUNT_COLLECTIONS`（账号） |

## 7. 未来换数据库（SQLite / PostgreSQL）

当前 JSON 文件方案适合轻量服务器 + 几十人规模。要换真数据库时：

- **只改一个文件**：`packages/server/src/infra/store.ts`，实现 `Store` 接口（`get/set/delete/all/clear`）。
- 领域模块、gateway、刷档命令**零改动**——它们只依赖 `Store` 接口，不关心底层是文件还是 DB。
- 集合名直接映射成表名，文档主键映射成主键列，文档 JSON 可整列存也可拆列。第 3 节的「集合/文档/主键」模型和关系型是一一对应的。

这正是架构把存储隔离在 infra 层的目的：换存储不动业务。
