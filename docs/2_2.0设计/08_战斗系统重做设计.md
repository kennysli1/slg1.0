## 一、总目标（一句话）

废掉"瞬时结算纯函数"，改成 **一个有状态的战斗流程**：战斗随时间逐 tick 减员，
势均力敌打得最久、一边倒最快；战斗进行中后到的部队按阵营立刻并入同一场战斗。

---

## 二、四条铁律怎么落（架构归属）

| 关注点 | 归属 | 说明 |
|--------|------|------|
| "进行中的战斗"这块状态 | **`combat.ts` 升级为有状态模块**（owner） | 不再是纯函数。owns `battle` 集合：每个进行中的战场一条记录。 |
| tick 减员的时间推进 | **Scheduler** | 每个战场登记一个"下一 tick"任务；禁止模块内 setInterval。 |
| 扣兵 / 归队 / 掠夺 / 战报 | 各 owner（military / economy） | combat 只算结果，发 Command 让 owner 改状态，发 Event 出战报。 |
| 兵种攻防/特性数值 | config（`units.csv` + 新 `unit_traits.csv`） | combat 读 config 拿静态数值，派生结果对外只给快照。 |

**关键变化**：原 `resolveCombat` 纯函数取消。战斗不再"一次算完"，而是"每 tick 算一小步"。
Movement 到达时不再自己调 combat 结算，而是发 Command 让 Combat **开一场战斗 / 并入已有战斗**。

## 三、兵种数据模型（units.csv 改列）

**彻底废弃 `cat`（兵种类别）概念**：不再有 infantry/cavalry/scout/siege/admin/settler
的区分，以及依附其上的一切特殊机制预期（侦察、攻城、拓荒建村、元老夺城——本就未实现，一并不做）。
兵种从此**只区分近战 / 远程**，其余靠攻防四列 + 特性表达。

**废弃列**：`atk`、`defInf`、`defCav`、`cat`。
**保留列**：`carry`（负重，胜利后按幸存者 carry 决定搬运战利品量，与战斗公式无关）、
`speed`（行军速度，属移动模型）、`upkeep`（耗粮）、`cost*`/`trainSec`/`building`（训练）。

**新增四列 + 一个形态 + 特性引用**：

| 列 | 含义 |
|----|------|
| `form` | 形态，`melee`(近战/前排) 或 `ranged`(远程/后排)。二选一。**取代 cat**。 |
| `meleeAtk` | 近战攻击力 |
| `rangedAtk` | 远程攻击力 |
| `meleeDef` | 近战防御力（被近战攻击时的承伤耐久） |
| `rangedDef` | 远程防御力（被远程攻击时的承伤耐久） |
| `traits` | 特性 id 列表（逗号分隔，引用 unit_traits.csv 的数字 id；可空） |

> 影响：`config.ts` UnitDef 删 cat 改四列；`meta.ts` 下发、`military.ts` 可训练列表、
> 前端 `info.ts`/`config.ts`/`army.ts` 的 cat 展示统一改为 form(近战/远程)；
> `combat.ts` 原骑兵加权逻辑随 cat 一起删除。

> 例：某近战步兵 `form=melee, meleeAtk=5, rangedAtk=0, meleeDef=40, rangedDef=20`。
> 某弓箭手 `form=ranged, meleeAtk=1, rangedAtk=10, meleeDef=15, rangedDef=15`。

### 新表 `config/unit_traits.csv`（特性 = 额外数值加成）

| 列 | 含义 |
|----|------|
| `id` | 数字主键（units.csv 的 traits 列引用它） |
| `code` | 英文代码（程序内部用，勿改） |
| `name` | 显示名（如"持盾"） |
| `effect` | 效果类型代码（枚举，见下） |
| `value` | 数值（含义随 effect 而定） |

初始 effect 枚举（可扩展）：
- `dmg_taken_ranged`：受远程伤害倍率（如 value=-0.30 → 受远程伤害 -30%，持盾）
- `dmg_taken_melee`：受近战伤害倍率
- `atk_ranged` / `atk_melee`：自身远程/近战攻击力加成倍率
- `def_ranged` / `def_melee`：自身对应防御加成倍率

> 特性是**在攻防四列之上的额外倍率修正**，不替代四列。加新特性只改 CSV + 在 combat 的 effect 分派里加一个 case。

## 四、战斗核心机制

### 4.1 阵营与站位

- **一地一场战**：同一目标格只有一个战场实体。以被攻击目标（玩家村 / PvE 点）为锚。
- **两个阵营**：`attacker`（所有来攻的部队合并）/ `defender`（守方驻军 + 后到的增援合并）。
- **两排**：每个阵营内部分 **前排 = form 为 melee 的所有兵**、**后排 = form 为 ranged 的所有兵**。

### 4.2 承伤顺序（谁先掉血）

**永远先掉前排（近战），前排全灭后才掉后排（远程）。**
即：只要本阵营还有活着的近战兵，本阵营的远程兵一兵不损。

### 4.3 每个单位这一 tick 用哪个攻击力

对**远程兵**（近战兵永远用 meleeAtk）：

| 己方近战存活 | 敌方近战存活 | 远程兵启用 |
|:---:|:---:|:---:|
| 有 | —— | `rangedAtk`（后排放） |
| 无 | 有 | `meleeAtk`（被迫接身肉搏） |
| 无 | 无 | `rangedAtk`（纯远程对射） |

> 边界：一方**从开局就没有近战兵**（纯远程军），等同于"近战全灭"——敌方若有近战冲上来，
> 该方远程兵从第一 tick 就用 meleeAtk 应战。

### 4.4 单 tick 减员公式（草案，待调参）

设 tick 间隔 `dt`（config，如 200ms），全局强度系数 `k`（config）。每 tick 对每个阵营：

1. **算火力**：按 4.3 决定每个存活兵此刻用 melee 还是 ranged 攻击力，
   - `meleeDmg_A = Σ (用近战的兵) count × meleeAtk ×(特性 atk_melee 修正)`
   - `rangedDmg_A = Σ (用远程的兵) count × rangedAtk ×(特性 atk_ranged 修正)`
2. **命中对方当前承伤排**（B 的前排；前排空了则后排）。把火力换算成击杀数：
   - 该排对近战的平均耐久 `mDef`（= 排内兵 meleeDef 均值×特性 def_melee 修正）
   - 该排对远程的平均耐久 `rDef`（同理，含持盾类 dmg_taken_ranged 修正）
   - `kills_B = k × dt × ( meleeDmg_A/mDef + rangedDmg_A/rDef )`
3. **按兵种比例把 kills 摊到该排各兵种**，扣减。
4. 两阵营**同时**用 tick 开始时的兵力互算（避免先手偏差）。

**为什么这个公式满足"势均力敌最久、一边倒最快"**：强的一方更快清掉对方承伤排 → 对方火力骤降 → 雪球效应加速 → 一边倒极快结束；双方接近时互相缓慢磨，持续最久。具体秒数由 `k`、`dt` 调参（放 game_constants.csv）。

### 4.5 结束条件

- **一方总兵力归零** → 战斗结束，另一方全胜。
- **输方不保留残兵**（打到归零），**不支持中途撤退/逃跑**（后置需求）。
- 胜方为 attacker → 触发掠夺（沿用现 economy.GetLootable/TakeLoot + 幸存者 carry 决定搬运量）+ 带战利品返程。
- 胜方为 defender → 守军保留幸存，attacker 全灭无返程。

### 4.6 中途并入

- 新部队到达时若该目标格**已有进行中的战场**：按其阵营（来攻=attacker / 增援=defender）
  把兵力并入对应阵营，**下一 tick 生效**（重算火力与承伤排）。
- 若无进行中战场：新开一场。
- **协防（本期做）**：Movement 新增 `reinforce` 类型——派兵去友方（自己或他人）村庄。
  到达时若该村正被攻打，援军并入 defender 阵营；若无战斗，则作为驻军停留（later 可撤回，本期可先只做并入/停留）。
- **PvE 合战（本期做）**：多个玩家同时攻打同一 PvE 点，全部并入 **attacker 阵营**（彼此不敌对），
  共同对 PvE 守军作战。战利品分配规则见开放问题。
