# 阶段三 · 技术栈与工程结构

> ✅ 技术栈已确认（2026-06-29）。本文记录选型 + 代码工程结构，作为写代码的依据。

## 技术栈（全栈 TypeScript）

| 层 | 选型 | 理由 |
|----|------|------|
| 语言 | **TypeScript**（前后端统一） | 信封/payload 类型定义两端共享一份，杜绝协议对不上 |
| 后端运行时 | **Node.js** | 与 TS 一体；事件驱动契合我们的调度器+事件总线架构 |
| 后端框架 | **Fastify** | 轻量，不绑架模块架构 |
| 实时通信 | **WebSocket** | 服务器需主动 Push（战斗结果/行军到达） |
| 主数据库 | **JSON 文件**（当前阶段） / PostgreSQL（成熟阶段） | 当前零依赖可部署；存储接口已隔离，成熟后可替换为结构化数据库 |
| 缓存/调度辅助 | **Redis**（后续按需） | 热数据/排行/定时任务的搭档，非主库，起步不上 |
| 前端 | **原生 TypeScript + Vite** | 零框架、启动快，当前按 feature 拆分渲染模块；后续需要复杂交互时可再引入框架 |

### 数据存储分阶段
```
测试阶段:  Persistence 层内部 = 内存（最快验证架构）
当前生产:  Persistence 层内部 = JSON 文件（零依赖、重启不丢）
成熟阶段:  Persistence 层内部 = PostgreSQL/SQLite（换内部实现，接口不变，其它模块零改动）
```
> Persistence 层隔离了存储细节，换数据库不影响游戏逻辑——这是架构"以后好改"的体现之一。

## 为什么是 PostgreSQL（备查）
- 关系型最适合"玩家→村庄→建筑/部队"强关联结构；自带事务保证"扣资源"原子。
- vs MongoDB：文档型弱事务弱关联，不适合我们。
- vs MySQL：PG 功能更强（JSON字段、坐标查询），更贴合"payload自由+地图查询"。
- Redis 是缓存搭档不是主库；CockroachDB 等分布式SQL解决的是千万级规模，现在用不上。

## 工程结构（monorepo，前后端共享类型）
```
slg1.0/
├── packages/
│   ├── shared/        ← 前后端共享：信封类型、payload类型、配置表类型
│   ├── server/        ← 后端
│   │   ├── infra/         基础设施层：Scheduler / EventBus / Persistence
│   │   ├── gateway/       接入层：WebSocket + Request↔Command / Event↔Push 翻译
│   │   ├── modules/       领域层：economy / building / military / movement / combat / world / pve / hero
│   │   └── main.ts        组装启动
│   └── client/        ← 前端原生 TS + Vite
└── docs/              ← 现有设计文档
```

## 工程约定（踩坑记录）

- **@slg/shared 必须编译成 ESM**（`"type":"module"` + exports 字段）。前后端共用：浏览器/Vite 只认 ESM 的具名导出，若 shared 编成 CommonJS（`export *` 经 `__exportStar`），Rollup 静态分析认不出导出 → 前端页面空白。前后端统一 ESM。
- **改动 shared 后必须 `npm run build:shared`**：前端/后端引用的是 `dist/`，不是源码。
- **端口占用排查**（Windows）：`netstat -ano | findstr ':8080'` 找 LISTENING 的 PID → `taskkill //F //PID <pid>`。正常停服在终端按 Ctrl+C 即可。
