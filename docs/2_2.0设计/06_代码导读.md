## 2. 三层架构（从下往上）

```
基础设施层 infra/    EventBus · CommandBus · Scheduler · Store
     ↑ 被领域层使用（注入，不反向依赖）
领域层 modules/      Player · Economy · Building · Military · World · PvE · Movement · Combat · Notifications
     ↑ 被 Gateway 调用
接入层 gateway/      Gateway（翻译） + main.ts（传输）
```

**依赖方向铁律**：上层依赖下层，下层永不依赖上层。模块之间**不互相直接 import**，只通过 CommandBus/EventBus 通信。

---

## 3. 两套信封（通信的全部）

### 边界① 服务器↔客户端（`shared/wire.ts`）
| 消息 | 方向 | 关键字段 |
|------|------|---------|
| `WireRequest` | 客户端→服务器 | `action` + `payload` |
| `WireResponse` | 服务器→客户端 | `ok` + `payload` + `error?` |
| `WirePush` | 服务器→客户端(主动) | `event` + `payload` |
公共信封：`v`(版本) `type` `id` `ts`。

### 边界② 模块↔模块（`shared/messaging.ts`）
| 类型 | 用途 | 信封 | 返回 |
|------|------|------|------|
| `Command` | 写操作/要确认 | `name` `from` `payload` | `CommandResult{ok,payload,reason}` |
| `DomainEvent` | 通知/解耦 | `name` `source` `ts` `payload` | 无 |

**口诀**：写操作走 Command（要成败），通知走 Event（要解耦）。payload 永远自由。

## 4. 一条链路的完整流动（升级建筑）

跟着代码走一遍，理解所有部件如何协作：

```
1. 前端 client/main.ts
   request('UpgradeBuilding', {villageId, kind})  → WS 发 WireRequest

2. server/main.ts  socket.on('message')
   → gateway.handleRequest(req)

3. gateway.ts
   查 ACTION_TO_COMMAND['UpgradeBuilding'] = 'building.TryUpgrade'
   → commandBus.send({name:'building.TryUpgrade', payload})

4. modules/building.ts  tryUpgrade()
   校验队列空位 → 算 cost
   → commandBus.send({name:'economy.TrySpend', payload:{villageId, cost}})

5. modules/economy.ts  trySpend()
   惰性结算 → 原子检查 → 够则扣 → 返回 {ok:true}

6. 回到 building.ts
   扣费成功 → scheduler.schedule(durationMs, completeUpgrade)
   → 返回 {ok:true, finishAt}  （经 Gateway 包成 WireResponse 回前端）

7. (时间到) scheduler 触发 completeUpgrade()
   等级+1 → eventBus.emit({name:'building.Upgraded', payload})

8. 两个订阅者各自响应：
   - economy.ts 订阅 building.Upgraded → 更新产率（派生管线加/调一层）
   - gateway.ts 订阅 building.Upgraded → 包成 WirePush 推给所有客户端

9. 前端收到 push 'BuildingUpgraded' → refresh() 刷新显示
```

> 这条链路把三铁律全用上了：状态各归各家（Building 不碰资源，发 Command 让 Economy 扣）、跨模块只传 payload、时间统一走 Scheduler。

## 5. 一个领域模块的标准结构（以 economy.ts 为例）

每个领域模块都长这个样子，照抄即可：

```ts
export class EconomyModule {
  static readonly NAME = 'economy';                 // 模块名

  constructor(
    private store: Store,                            // 注入基础设施（不自己new）
    private bus: EventBus,
    private commands: CommandBus,
    private now: () => number,                        // 时间源也注入（测试可换假时钟）
  ) {}

  init(): void {                                      // 注册对外能力
    this.commands.register('economy.TrySpend', c => this.trySpend(c));   // 提供 Command
    this.bus.on('building.Upgraded', evt => this.onBuildingUpgraded(...)); // 订阅 Event
  }

  // 私有方法实现各 Command/Event 的逻辑，只读写自己的 store 集合
}
```

要点：
- **依赖注入**：infra 和 now 都从构造函数传入，模块不自己 `new`。便于测试替换（假时钟、内存 store）。
- **init() 集中注册**：所有对外的 Command 处理器和 Event 订阅都在 init() 里，一眼看全这个模块"提供什么、关心什么"。
- **只碰自己的 store 集合**：economy 只读写 `'economy'` 集合，绝不去读 `'building'` 集合。

## 6. 测试怎么写

- **端到端单测**（`upgrade-flow.test.ts`）：用 `createGameApp({manualScheduler:true, now:假时钟})` 组装，`scheduler.advanceTo()` 快进时间，断言状态。**不连网络、不等真实时间**。

## 7. 跑起来 / 常用命令

```bash
npm install                      # 首次
npm run build:shared             # 改过 shared 后必须重跑

npm run dev:server               # 后端 :8080
npm run dev -w @slg/client       # 前端 :5173（自动代理 /ws 到后端）

npm run test:server              # 后端测试
```

> 改了 `shared/` 一定要 `build:shared`，否则前后端用的是旧的 `dist/`。
