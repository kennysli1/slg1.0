## 〇、一句话结论

**建筑系统仍是单一 owner：重写（而非拆分）`modules/building.ts`。** 三区（center/inner/outer）、资源田、建造队列都是同一块"每村建筑布局"状态的组成部分，只有一个 owner。城内/城外**槽位数、建造队列条数是从城镇中心等级派生的快照**（铁律#4），不是独立状态，因此不需要独立模块。`fields.csv` 废弃并入 `buildings.csv`。此外，借这次重构把散在 economy 里的"建筑→经济"派生（仓储容量公式）收回 building 内部，修掉一处既有的铁律#1/#4 渗漏。

## 一、Step 1 · 需求分析

### 1.1 新增/变化的状态（谁拥有）

| 状态 | 说明 | owner |
|------|------|-------|
| 城镇中心等级 | 唯一成长阀门 | Building（作为 zone=center 的一个 placed 实例） |
| 城内/城外已建建筑实例 | 每个实例有 slotId / zone / kind / level | Building |
| 资源田实例 | 合并为 zone=outer 且带产出字段的建筑实例 | Building |
| 建造队列 | 从单条改为多条（开局 2，随 TC 增长） | Building |
| **城内/城外槽位数** | **派生自 TC 等级**（`town_center_slots.csv`），非独立状态 | Building 内部计算，对外给快照 |
| **建造队列容量** | **派生自 TC 等级**，非独立状态 | Building 内部计算 |

**结论**：无任何"新的独占状态块"需要新 owner。全部归 Building。按扩展决策树总闸——说不出新 owner → 不加模块，重写既有 owner 文件。

### 1.2 读/改的既有状态

- **Economy**（owner=economy）：建造/升级扣资源，走既有 `economy.TrySpend`。资源田产出走既有 `economy.SetBaseRate`。人口耗粮走既有 `economy.SetUpkeep`。**仓储容量**目前 economy 通过订阅 `building.Upgraded` 自行按公式算——本次改为 building 计算后经新 Command 上报（见 §5）。
- **Military**（owner=military）：`smithyLevel`（兵种攻防养成）由 military 自持，与建筑表里的 `smithy` 建筑是两回事。建筑只提供 `smithy` 作为前置里程碑，military 侧不动。军事建筑（兵营/马厩/工坊/校场）是否已建、等级多少，由 military 在训练校验时向 building 查询快照（当前 military 未校验建筑，二期若要"必须有兵营才能训兵"再加，见 §5.3）。
- **Combat**（owner=combat）：结算时读城墙等级。当前直接 `building.GetState` 后挖 `buildings.wall`——本次改为 building 暴露聚合防御快照命令（见 §3.2、§5.2）。

### 1.3 跨模块交互清单

| 方向 | 交互 | 信封 | 现状/新增 |
|------|------|------|-----------|
| Building → Economy | 建造扣费 | `economy.TrySpend` | 现有，复用 |
| Building → Economy | 资源田总产率上报 | `economy.SetBaseRate` | 现有，复用（改为遍历 placed 里 zone=outer 且有产出的实例） |
| Building → Economy | 人口耗粮上报 | `economy.SetUpkeep` | 现有，复用 |
| Building → Economy | **仓储容量上报** | `economy.SetCapacity` | **新增**（把公式从 economy 收回 building） |
| Building → (broadcast) | 建筑升级完成 | `building.Upgraded` | 现有，复用（notifications/economy 订阅） |
| Building → (broadcast) | **新建筑落成** | `building.Built` | **新增**（可选，用于通知/前端刷新区分"新建 vs 升级"） |
| Combat → Building | 取城墙防御快照 | `building.GetDefenseSnapshot` | **新增**（替代 combat 挖 raw 状态） |

### 1.4 客户端影响

- 村庄页从"资源田网格 + 中心建筑网格"两段式，改为**三区视图 + 空槽点击 → 侧边栏可建列表 → 建造**。
- 客户端不持有权威状态，全部渲染服务端 `Push`/`Response` 快照。槽位空满、可建灰显理由等全部由服务端算好下发。

### 1.5 朴素做法会踩的铁律雷（预警）

- ❌ **让 Combat / Military 直接读 building 的 `placed` 数组判断城墙/兵营** → 违反铁律#1。必须走命令拿快照。
- ❌ **让 Economy 继续按 building 的等级套公式算仓储容量** → 既有渗漏，多仓库后还会算错。改为 building 算好总量上报（铁律#4）。
- ❌ **把"槽位数"落库成状态字段** → 冗余且会与 TC 等级不一致。它是纯派生，每次从 config 现算。
- ❌ **多条建造队列用 `setTimeout` 各自计时** → 违反铁律#3。每条队列一个 `scheduler.schedule`，taskId 存进队列项，`resume()` 重登记。

## 二、Step 2 · 架构决策

### 2.1 模块边界：重写 Building，不拆分

单一 owner 原则下，Building 继续独占 `building` 集合。内部状态结构从"扁平 buildings map + fields 数组"升级为"槽位化的 placed 实例数组 + 多条队列"。

**新状态结构草案（仅示意，落地时定稿）：**

```ts
type Zone = 'center' | 'inner' | 'outer';

interface PlacedBuilding {
  slotId: string;   // 村内唯一，如 'inner-0' / 'outer-3' / 'center'
  zone: Zone;
  kind: string;     // 建筑 code（含资源田，如 woodcutter）
  level: number;
}

interface QueueItem {
  slotId: string;         // 目标槽（新建时先占位一个待建 slotId）
  kind: string;
  toLevel: number;        // 新建=1，升级=当前+1
  isNew: boolean;         // 新建 vs 升级（决定完成时 push building.Built 还是 Upgraded）
  startAt: number;
  finishAt: number;
  taskId: string;         // Scheduler 任务句柄
}

interface BuildingState {
  villageId: string;
  tribe: string;
  placed: PlacedBuilding[];   // 含 center/inner/outer 全部已建实例
  queue: QueueItem[];         // 多条并行，容量派生自 TC 等级
}
```

> **为何 center 也放进 placed？** 让 TC 成为普通 placed 实例（slotId='center'、zone='center'、kind='main'），升级逻辑与其它建筑同构，只是它的 level 会被用来派生槽位数与队列容量。避免为 TC 单开一套特殊字段。

### 2.2 派生（全部在 Building 内部算，对外给快照 — 铁律#4）

- `innerSlots(tcLevel)` / `outerSlots(tcLevel)` / `queueCapacity(tcLevel)`：查 `town_center_slots.csv`。
- 某区 `freeSlots = slots(zone) - count(placed in zone)`。
- 某区"可建建筑列表"：遍历 `config.buildings` 中 `zone===该区` 的定义，附 `unlocked`（requires 是否满足）+ 缺失前置说明 + 造价 + 耗时。
- 全村资源田总产率、总人口耗粮、总仓储容量：遍历 placed 聚合。

### 2.3 建造队列并发模型

- 队列改为数组，长度上限 = `queueCapacity(tcLevel)`（开局 2，随 TC 里程碑 +1，配在 `town_center_slots.csv` 的 `queueSlots` 列）。
- 每个 `QueueItem` 各自 `scheduler.schedule(delay, ...)`，完成回调只结算自己那条并从数组移除。`resume()` 遍历所有队列项逐条重登记（沿用现有 resume 模式）。
- 新建建筑入队时即在 `placed` 里插入一个 level=0 的占位实例（占住 slot 防并发超建），完成时 level→1 并 push `building.Built`；也可选择完成时才落 placed（则占位信息只存在 queue 项里）。**倾向前者**：点击空槽建造后该槽立即显示"建造中"，符合交互预期。

### 2.4 配置优先

新增建筑/资源田/调槽位曲线**只改 CSV**。zone、产出字段、槽位曲线全部配置化。requires 沿用建筑数字ID:等级机制（`main:5` 之类）。

## 三、Step 3 · API 契约

### 3.1 服务器 ↔ 客户端（Request / Response / Push）

沿用 manifest 声明式注册（在 `BuildingModule.MANIFEST.publicActions` 加行即可）。

| Action（对外） | 内部 Command | ownVillage/auth | Response payload 快照 |
|---|---|---|---|
| `GetVillageLayout` | `building.GetLayout` | ✓/✓ | 见下 A |
| `GetBuildOptions` | `building.GetBuildOptions` | ✓/✓ | 见下 B |
| `Build`（点空槽建造） | `building.Build` | ✓/✓ | `{ slotId, kind, finishAt }` |
| `UpgradeBuilding`（点已建升级，含资源田） | `building.Upgrade` | ✓/✓ | `{ slotId, toLevel, finishAt }` |

> **废弃**：旧 `UpgradeField`（资源田并入 `UpgradeBuilding`，用 slotId 定位，不再用 fieldIndex）。旧 `GetVillage` 由 `GetVillageLayout` 取代。

**A. `GetVillageLayout` 响应（村庄总快照）：**
```ts
{
  townCenter: { slotId: 'center', kind: 'main', level: number,
                nextCost, nextTimeSec, maxLevel },
  zones: {
    inner: { slots: number, placed: PlacedView[], freeSlots: number },
    outer: { slots: number, placed: PlacedView[], freeSlots: number },
  },
  queue: { capacity: number, items: QueueView[] },
}
// PlacedView：{ slotId, kind, name, icon, level, maxLevel,
//               nextCost|null, nextTimeSec|null, producing?: {resource, ratePerHour} }
// QueueView：{ slotId, kind, name, toLevel, isNew, startAt, finishAt }
```

**B. `GetBuildOptions` 请求 `{ zone }` → 响应（该池可建清单，给侧边栏）：**
```ts
{
  zone: 'inner' | 'outer',
  freeSlots: number,
  options: Array<{
    kind, name, icon,
    cost, timeSec,               // level 1 造价/耗时
    unlocked: boolean,           // requires 是否满足
    requires: { kind, level }[], // 前置（前端灰显文案用）
    lockReason?: string,         // 如 "需城镇中心 5 级"
  }>,
}
```

**Push（eventPushMap）：**
| 内部事件 | 对外推送 | 触发 |
|---|---|---|
| `building.Upgraded` | `BuildingUpgraded` | 升级完成（保留，兼容 notifications/economy） |
| `building.Built` | `BuildingBuilt` | 新建筑落成（新增；payload 含 villageId、slotId、kind） |

> 派生只给快照：Response 不下发 requires 公式/成本函数，只给算好的 cost/timeSec/unlocked/lockReason（铁律#4）。

### 3.2 模块 ↔ 模块（Command / Event）

**Building 对外新增：**
- `building.GetDefenseSnapshot { villageId } → { wallLevel }`（供 Combat 用；替代其直接挖 `GetState().buildings.wall`）。是聚合防御快照，二期可扩展加保护半径等，不改调用方。

**Building 对 Economy 新增调用：**
- `economy.SetCapacity { villageId, capacity: Partial<ResMap> }`（Building 遍历 placed 里所有 warehouse/granary 算出总容量后上报；Economy 只存不算）。

**Economy 侧改动：**
- 删除 `onBuildingUpgraded` 里 `warehouse`/`granary` 分支的容量公式；改为注册 `economy.SetCapacity`。`storage_base` / `storage_growth_per_level` 常量的消费方从 economy 移到 building。
- `building.Upgraded` 订阅可保留（若还有其它用途）或清理，视 economy 是否还需感知。

**命名规范核对**：命令祈使式（`building.Build` / `building.Upgrade` / `economy.SetCapacity`）、事件过去式（`building.Built` / `building.Upgraded`），均带模块命名空间。✓

## 四、Step 2 补充 · 配置层改动

### 4.1 `buildings.csv` 新增列

在现有列基础上新增（放在 `code` 后、成本列前，便于阅读）：

| 新列 | 取值 | 说明 |
|---|---|---|
| `zone` | `center` / `inner` / `outer` | 归属区。资源田填 `outer` |
| `resource` | wood/clay/iron/crop 或空 | 仅资源田填；非产出建筑留空 |
| `prodBase` | 数字或空 | 仅资源田：每级产量基数（迁移自 fields.csv） |
| `prodGrowth` | 数字或空 | 仅资源田：产量增长率（迁移自 fields.csv） |

**合并后 `buildings.csv` 列头（草案）：**
```
id,code,name,icon,zone,resource,prodBase,prodGrowth,costWood,costClay,costIron,costCrop,costGrowth,timeBase,timeGrowth,maxLevel,requires
```
- 现有 10 栋建筑补 `zone`：main→center；warehouse/granary/academy/smithy/wall/market→inner；barracks/stable/workshop/rallypoint→outer。
- 追加 4 行资源田（zone=outer，填 resource/prodBase/prodGrowth）：woodcutter/claypit/ironmine/cropland，参数从 `fields.csv` 搬来。
- `fields.csv` **删除**（策划已定废弃）。

### 4.2 新增 `town_center_slots.csv`

```
tcLevel,innerSlots,outerSlots,queueSlots,unlockNote
1,2,3,2,开局
2,...
...
20,10,18,4,满级
```
- **逐级写全 1..20**（策划待定项3 倾向逐级写全，避免插值歧义）。中间级别数值由数值同学补，本表结构先定。
- `queueSlots`：建造队列容量随 TC 增长（待定项4 倾向"是"）。若数值同学决定不做，则全列填 2。

### 4.3 `village_templates.csv` 改动

- `field_layout` 语义变为"开局在城外区预置的资源田实例"，`start_buildings` 变为"开局各区预置建筑（含 center 的 main）"。二者都要能表达 zone + slot 归属。
- **建议**：合并为一列 `start_placed`，格式 `code:level@zone`，如 `main:1@center|woodcutter:0@outer|woodcutter:0@outer|...`。或保留两列但解析时按 buildings.csv 的 zone 自动归区（更省事，推荐后者——布局里只写 code:level，zone 从建筑定义查）。
- 开局预置的资源田数量不能超过 tcLevel=1 的 outerSlots，否则校验报错（见 4.5）。

### 4.4 `infra/config.ts` 类型/加载改动

- `BuildingDef` 增字段：`zone: Zone`、`resource?: string`、`prodBase?: number`、`prodGrowth?: number`。
- 删除 `FieldDef` 与 `fields` 的加载分支及 `GameConfig.fields`（下游改读 buildings 里 zone=outer+有 resource 的项）。**注意连带改**：economy 校验、meta 下发、client config.ts 的 fields 处理。
- 新增 `GameConfig.townCenterSlots`：`{ inner: number[]; outer: number[]; queue: number[] }` 或 `Record<number, {inner,outer,queue}>`，附取值辅助函数（给定 tcLevel 返回三值，缺级时报错或就近）。
- `loadGameConfig` 增 `town_center_slots.csv` 解析。

### 4.5 `validateGameConfig` 增补校验

- 每栋建筑 `zone` ∈ {center,inner,outer}；恰好一栋 zone=center。
- zone=outer 且填了 resource 的，resource 必须在 resources.csv；prodBase≥0。
- `town_center_slots.csv`：tcLevel 覆盖 1..main.maxLevel；slots 单调不减；queueSlots≥1。
- village_templates 开局预置数量 ≤ tcLevel=1 各区 slots。
- 原有 requires 循环检测等保留。

## 六、存档不兼容与刷档需求（明确）

**本次为破坏性结构变更，存档不兼容，必须刷档。**

原因：
1. `building` 集合的文档结构从 `{buildings:map, fields:[]}` 变为 `{placed:[], queue:[]}`——旧存档无法反序列化为新结构。
2. 资源田从独立 fields 数组迁入 placed，slotId 体系全新。
3. `fields.csv` 废弃，旧存档里引用的 field 定义可能不存在。

**处理方式**：不写迁移脚本（骨架期，成本不划算），直接刷档。
- 本地/测试：`npm run reset:season`（留账号+地图位置，进度归零）。
- 部署：`bash .claude/deploy/deploy.sh --reset season`（数据结构变更规范要求带刷档）。

**校验点**：`economy` 集合结构未变（仍存 resources/baseRate/capacity 等），但因 building 重建会重新上报 baseRate/capacity/upkeep，刷档后首个 `createVillage` 即写入正确值，无需迁移 economy。

## 七、风险 & 铁律核验清单

| 风险点 | 对策 |
