# 阶段二 · 通信格式规范

> ✅ **已定稿（2026-06-29，用户确认）**。
> 本文只定**通信的"信封"格式**（外层固定结构），不规定每条消息的具体内容。
> 内容(payload)随业务自由变——升级只放资源、战斗放资源+兵力——这些会变，不在这里锁。
> 锁死的是：两个边界各自的信封长什么样、有哪几类消息、字段含义。**信封稳定 = 协议稳定；payload 灵活 = 业务可扩展。**

---

## 核心原则

> **固定外层信封，自由内层 payload。**
> 所有消息都套同一个信封（`type` + `payload` + 元数据）；接收方先看 `type` 决定怎么处理，`payload` 的结构由 `type` 自己定义、可随版本演进。新增一种业务消息 = 加一个新 `type`，不动信封、不动已有消息。

两个边界，两套信封：

```
边界①  框架外：服务器 ↔ 客户端   →  网络协议信封（要跨网络、要兼容多端、轻易不改）
边界②  框架内：模块 ↔ 模块        →  内部消息信封（Command / Event，进程内或跨服务）
```

---

## 边界① · 框架外：服务器 ↔ 客户端

跨网络通信。三类消息，统一信封。

### 通用信封字段
| 字段 | 含义 | 所有消息都有 |
|------|------|------|
| `v` | 协议版本号 | ✅ 兼容演进用 |
| `type` | 消息类型（决定 payload 结构） | ✅ |
| `id` | 消息唯一id（请求-响应配对、去重） | ✅ |
| `ts` | 时间戳（服务器时间，客户端对齐倒计时） | ✅ |
| `payload` | 业务内容（结构由 type 定义，自由） | ✅ |

### 三类消息

**1. Request（客户端→服务器：我要做某事/要数据）**
```
{ v, type:"req", id, ts, action:"<动作名>", payload:{...自由...} }
   action 例：UpgradeBuilding / TrainTroops / SendMarch / GetVillage ...
```

**2. Response（服务器→客户端：对某个 Request 的应答）**
```
{ v, type:"res", id:<对应req的id>, ts, ok:true/false, payload:{...} , error?:{code,msg} }
   ok=false 时 error 说明原因（如 资源不足/前置不满足）
```

**3. Push（服务器→客户端：主动推送，无需客户端先问）**
```
{ v, type:"push", id, ts, event:"<事件名>", payload:{...自由...} }
   event 例：BuildDone / UnderAttack / MarchArrived / ResourcesChanged ...
```

### 关键约定（要稳定的部分）
- **C1 信封字段固定**：`v/type/id/ts` 永远在，payload 永远是个对象。客户端按这个解析，不依赖具体 action/event 也能收发。
- **C2 时间用服务器 ts**：所有倒计时（建造剩余、行军到达）= 客户端用 `结束时刻 - 本地估算的服务器now` 自行推算，**不靠服务器每秒推送**（呼应惰性结算思想）。
- **C3 版本演进**：加字段只往 payload 里加；删/改字段升 `v`。老客户端忽略不认识的字段。
- **C4 鉴权/会话**：放在连接握手层（不在每条消息里重复带 token，由连接本身代表身份）。

## 边界② · 框架内：模块 ↔ 模块

进程内（或未来拆服务时跨进程）。两类消息：**Command**（要确认结果）和 **Event**（广播通知）。

### Command（A 模块请 B 模块做事，等返回）
```
Command 信封:  { name:"<命令名>", from:"<模块>", payload:{...自由...} }
Result 信封:   { ok:true/false, payload:{...}, reason?:"<失败原因>" }
```
- 用于**写操作**和**需要立即知道成败**的场景。
- 例：`TrySpend{ village, cost:{...自由：升级时只有资源；别处可加别的} }` → `{ok, reason}`。
- payload 内容随命令变，**信封 `name/from/payload` 三件套固定**。

### Event（某模块状态变了，广播；订阅者各自响应）
```
Event 信封:  { name:"<事件名>", source:"<模块>", ts, payload:{...自由...} }
```
- 用于**通知/解耦**，发布者不关心谁在听。
- 例：`BuildingUpgraded{ village, building, level }`、`CombatResolved{ ...自由：可含损失兵力+掠夺资源+拆毁建筑 }`。
- **新增模块（如未来工会）= 订阅已有 Event + 发自己的 Event**，不改任何人。

### 关键约定
- **I1 信封固定**：Command 是 `name/from/payload`，Event 是 `name/source/ts/payload`。payload 自由。
- **I2 写操作走 Command（原子、要确认），通知走 Event（异步、解耦）**。
- **I3 不传内部对象引用**，只传 payload（值/快照）。这是"模块不读彼此内部状态"的落地。
- **I4 时间统一**：需要延时触发的，不在模块里自己计时，而是登记到 Scheduler，到点由 Scheduler 发一条 Event/调一个 Command。

## 两个边界怎么衔接（一眼看全）

```
客户端 ──Request(req信封)──▶ Gateway ──翻译成──▶ Command(内部信封) ──▶ 领域模块
客户端 ◀──Response(res信封)── Gateway ◀──Result──── 领域模块
客户端 ◀──Push(push信封)──── Gateway ◀──订阅Event── 领域模块(EventBus)
```
- **Gateway 是唯一的翻译官**：把外部 Request 翻成内部 Command；把内部 Event 翻成外部 Push。
- 两套信封因此**互相隔离**：改内部消息不影响客户端协议，改客户端协议不影响内部模块。

---

## 这份规范锁死了什么、留活了什么

| 锁死（稳定性） | 留活（可扩展性） |
|--------------|----------------|
| 两套信封的外层字段 | 每条消息的 payload 内容 |
| 三类外部消息(req/res/push) + 两类内部消息(Command/Event) | action/event/command 的种类（随便加新的） |
| Gateway 作为唯一翻译官 | 新增模块、新增业务消息 |
| 时间走服务器ts + Scheduler | 具体业务怎么算 |

> 升级=payload 里只放资源；战斗=payload 里放资源+兵力+建筑——**都是同一个信封，换 payload 而已**。这正是你要的"通信格式定死、内容可改"。
