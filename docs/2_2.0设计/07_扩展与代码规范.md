# 扩展与代码规范（Conventions）

> 立规矩，让"加东西"有章可循、不破坏架构。任何人（你/我/未来协作者）加代码前先读这份。
> 配套：代码现状见 `06_代码导读.md`，架构原理见 `03_架构总览.md`。

---

## 一、四条不可违背的架构铁律

> 违反任何一条 = 埋下"以后改不动"的雷。Code review 第一关就查这四条。

1. **状态归属唯一**：每块数据只有一个模块是 owner。别的模块要读/改，只能通过该模块的 Command，**绝不能直接 import 它的 store 集合或内部类型**。
2. **跨模块只传 Command/Event**：模块之间不互相 `import`、不互相调用方法。要协作，发 Command（要结果）或 Event（广播）。
3. **时间统一走 Scheduler**：任何"未来要发生的事"都 `scheduler.schedule()`，模块内**禁止 `setTimeout/setInterval`**。
4. **派生属性对外只给结果快照**：养成/加成在模块内部分层叠加，对外只暴露算好的最终值，不暴露怎么算的。

## 二、扩展决策树：我要加的东西属于哪类？

> **先问总闸这一句，就知道"加文件还是改文件"：**
> **我要加的东西，有没有"自己独占的一块状态"（比如工会成员表、邮件收件箱、任务进度）？**
> - **有** → 它需要一个 owner，**新建模块文件** `modules/xxx.ts`（见 §四）。一个模块文件 = 一块状态的唯一 owner，这正是铁律#1。
> - **没有**（只是给已有状态加数值/加成/触发效果）→ **改已有文件**：配置表 / owner 模块的派生管线 / 流水线插件。
>
> 换句话说：**能一口说出"这块新状态的 owner 是谁"→ 加文件；说不出、只是让旧状态多一点花样 → 改文件。** 下面的决策树是这句话的展开。

```
我要加的东西……
│
├─ 是"内容/数值"（新建筑、新兵种、新资源、新PvE目标、新装备）？
│     → 改【配置表】，不写新逻辑。见 §三
│
├─ 是"全新的一块状态"（工会、邮件、任务系统）？
│     → 加一个【新领域模块】。见 §四
│
├─ 是"养成/加成，影响已有系统的数值"（天赋、突破、buff）？
│     → 在 owner 模块的【派生管线加一层】。见 §五
│
└─ 是"战斗中动态触发的效果"（开战眩晕、血量触发狂暴）？
      → 写一个【Combat 流水线步骤插件】。见 §六（Combat 实现后）
```

> 先归类，再动手。绝大多数扩展是前两类。

## 三、加"内容/数值"——改配置表（CSV）

**原则：内容是数据，不是代码。** 所有游戏数值都在仓库根 `config/*.csv`，双击用 Excel 编辑，改完重启后端生效，**不改代码、不重新编译**。

现状配置表（见 `config/README.md`）：
| CSV | 内容 |
|-----|------|
| resources.csv | 资源种类 |
| fields.csv | 4 类资源田（产量/成本/时间公式参数） |
| buildings.csv | 中心建筑（成本/时间/前置依赖） |
| units.csv | 兵种（三维/速度/载货/耗粮/成本） |
| pve_targets.csv + pve_defenders.csv | PvE 目标模板与守军 |
| pve_spawns.csv | PvE 在地图的分布点 |
| game_constants.csv | 全局常量/平衡参数（城墙/铁匠加成、容量公式、地图尺寸…） |
| village_templates.csv | 各部族开局布局（田地/初始建筑/初始资源） |

**规范**：
- 加建筑/兵种/资源/PvE = 在对应 CSV 加一行，**不写新逻辑**。**前端也不用改**——名称/图标/分类由服务端 `GetGameConfig` 下发，客户端 `app/config.ts` 缓存消费（`info.ts` 仅作回退）。
- 成本/时间用"基数 × 增长率^(等级-1)"参数化，CSV 存基数和增长率两列。
- 代码侧由 `infra/config.ts` 的 `loadGameConfig()` 统一解析成 `GameConfig`，并在启动时 `validateGameConfig()` 校验（跨表引用/范围/循环依赖），注入各模块。模块从 `this.config.*` 读，**绝不 hardcode 数值或内容名**。
- 平衡参数（原先散在代码里的 magic number）统一进 `game_constants.csv`；新增常量须在 `config.ts` 的 `GameConstants` 加字段。
- 加一种**全新维度**的配置（如新增一张"科技表"）= 在 `config/` 加 CSV + 在 `config.ts` 加解析 + 补一条校验。

## 四、加"新领域模块"——照模板抄

新模块（如未来 Military/Movement/Combat/Guild）一律照这个骨架，放到 `server/src/modules/<name>.ts`：

```ts
import type { Command, CommandResult } from '@slg/shared';
import type { Store } from '../infra/store.js';
import type { EventBus } from '../infra/event-bus.js';
import type { CommandBus } from '../infra/command-bus.js';
import type { ModuleManifest } from '../gateway/manifest.js';
// 需要定时就再注入 Scheduler

interface XxxState { /* 本模块独占的状态结构 */ }
const COLLECTION = 'xxx';                          // 自己的 store 集合名，别人不碰

export class XxxModule {
  static readonly NAME = 'xxx';

  // 对外清单：网关启动时汇总它生成路由（自描述，免去手改 gateway）
  static readonly MANIFEST: ModuleManifest = {
    moduleName: 'xxx',
    publicActions: {
      // 外部动作名 → 内部命令；ownVillage=作用于自己村(强制注入会话villageId)，needAuth=需登录
      DoSomething: { command: 'xxx.DoSomething', ownVillage: true, needAuth: true },
    },
    eventPushMap: {
      // 内部事件 → 对外推送名（payload 须含 villageId 用于定向投递）
      'xxx.SomethingHappened': 'XxxHappened',
    },
  };

  constructor(
    private store: Store,
    private bus: EventBus,
    private commands: CommandBus,
    private now: () => number,
  ) {}

  init(): void {
    // 1. 注册对外提供的 Command（命名空间：'模块名.动作'）
    this.commands.register('xxx.DoSomething', c => this.doSomething(c));
    // 2. 订阅关心的 Event
    this.bus.on('other.SomethingHappened', evt => this.onThat(evt));
  }

  private doSomething(cmd: Command): CommandResult {
    // 只读写自己的 COLLECTION；要别的模块的数据/动作 → commandBus.send
    return { ok: true, payload: {} };
  }
}
```

**接线步骤**（加模块后必做）：
1. 在 `app.ts` 的 `createGameApp()` 里 `new XxxModule(...)` 并 `.init()`，挂到返回的 GameApp 上。
2. 若要对客户端开放：在模块上写好 `static MANIFEST`，再到 `gateway.ts` 的 `MODULE_MANIFESTS` 数组里登记 `XxxModule.MANIFEST` —— 路由表自动汇总生成，**不必再手改 ACTION_ROUTES/EVENT_TO_PUSH**。动作/事件名若与已有冲突，启动即报错。
3. 写一个端到端测试，并在 `src/test/all.test.ts` barrel 里加一行 `import './xxx.test.js'`。

**硬性规则**：
- 模块名小写，Command/Event 名用 `模块名.动词` 命名空间（`economy.TrySpend`、`building.Upgraded`），避免撞名。
- 一个 Command 名只能有一个处理器（CommandBus 重复注册会报错，这是故意的）。
- Event 名用过去式（已发生）：`building.Upgraded`、`combat.Resolved`。
- Command 名用祈使/尝试式（请求做）：`building.TryUpgrade`、`economy.Grant`。

## 五、加"养成/加成"——派生管线加一层

不要在战斗或产出公式里塞养成的 if-else。养成是 owner 模块属性计算里的**一层 modifier**。

现状示例（`economy.ts` 的产率管线）：
```ts
private finalRate(s, t): number {
  let mult = 1;
  for (const m of s.rateModifiers) mult += m.mult[t] ?? 0;  // 每个来源是一层
  return s.baseRate[t] * mult;
}
```
加工会产量 buff = 往 `rateModifiers` push 一层 `{source:'guild', mult:{wood:0.1}}`，并在 init() 订阅 `guild.BuffChanged` 事件来增删这层。**finalRate 的代码一行不用改。**

**规范**：
- 每个能被养成影响的属性，都要有"基础值 + modifier 列表"的结构。
- 新增 buff 来源 = push 一层 modifier + 订阅一个事件。
- modifier 带 `source` 标识，便于增删/调试。

## 六、加"战斗触发效果"——流水线步骤插件

（Combat 模块实现后补充细则。）原则：Combat 是无状态纯函数，由若干"步骤"组成流水线（侦察→主战→拆墙→掠夺…）。战斗中触发的技能 = 往流水线插一个步骤，**不改主流程**。

## 七、代码风格规范

- **语言**：TypeScript，`strict` 全开。不用 `any`（万不得已加注释说明）。
- **模块格式**：全项目 ESM（`import ... from '...js'`，相对导入带 `.js` 后缀——NodeNext 要求）。
- **命名**：类 `PascalCase`，函数/变量 `camelCase`，常量配置表 `UPPER_SNAKE`，文件 `kebab-case.ts`。
- **注释**：每个文件顶部一段块注释，写明"职责 + 对应哪份设计文档"。公开方法写简短用途。注释解释**为什么**，不复述代码。
- **中文**：注释和文档用中文（团队习惯），标识符用英文。
- **错误处理**：Command 失败返回 `{ok:false, reason:'类别:细节'}`（如 `insufficient:wood`），不抛异常穿透模块边界。
- **不过度设计**：骨架阶段够用即可，但不能违反 §一铁律。可扩展性靠铁律保证，不靠提前堆抽象。

## 八、提交前自查清单

> 前 4 条现在有**自动闸门**兜底，写错会当场红线或测试挂——但仍建议自查，闸门只覆盖能机械判断的部分。
> · 写代码时：装 ESLint 插件，跨模块 import / 模块内 setTimeout 会实时红线。
> · 提交前：`npm run lint`（铁律#2、#3）+ `npm run test:server`（含 `architecture.test.ts` 兜底铁律#1、#2、#3）。

- [ ] 先过了 §二"总闸"：新东西有独占状态→加模块文件，没有→改已有文件？
- [ ] 没有跨模块直接 import（只用 Command/Event，combat 纯函数除外）？ ← lint + 架构测试守
- [ ] 没有模块内 `setTimeout/setInterval`（用 Scheduler）？ ← lint + 架构测试守
- [ ] 没有别的模块直接 `store.*('别人的集合')`？ ← 架构测试守（铁律#1）
- [ ] 新内容是加到配置表，而非 hardcode if-else？
- [ ] 养成/加成是派生管线加层，而非塞进公式？
- [ ] Command/Event 命名带模块命名空间、时态正确？
- [ ] 改了 shared 后跑了 `build:shared`？
- [ ] 加了对应的端到端测试，且通过？
- [ ] 文件顶部注释写了职责和对应设计文档？
- [ ] `npm run lint` 与 `npm run test:server` 都绿？
