## 一、架构要解决的核心问题

回顾你的诉求：避免"AI 一把梭写死、系统间耦合不清、后续难改"。所以架构的第一原则是：

> **模块之间只通过明确的契约通信，不直接读写彼此的内部状态。**

具体落成三条铁律（所有模块都遵守）：

1. **状态归属唯一**：每块数据只有一个模块是它的 owner，别人要改只能通过 owner 暴露的接口。
2. **跨模块只传两种东西**：**命令(Command)**（"我请求做某事"）和**事件(Event)**（"某事已发生"）。不传内部对象引用。
3. **时间驱动统一**：所有"未来发生的事"都登记到一个统一的调度器，模块不自己藏定时器。

## 二、模块切分

按"状态归属"切分，每个模块管一块明确的状态。分三层：

```
┌─────────────────────────────────────────────────────────────┐
│  接入层 (Gateway)                                             │
│  - 客户端连接、鉴权、把请求翻译成 Command、把 Event 推给客户端  │
└───────────────────────────┬─────────────────────────────────┘
                            │ Command ↓        Event ↑
┌───────────────────────────▼─────────────────────────────────┐
│  领域层 (Domain Modules) — 游戏逻辑，每个模块管一块状态          │
│                                                               │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐         │
│  │ Economy  │ │ Building │ │ Military │ │ Movement │         │
│  │ 资源经济  │ │ 建筑科技  │ │ 军队兵种  │ │ 行军     │         │
│  │ (A组)    │ │ (B组)    │ │ (C组)    │ │ (D组前半) │         │
│  └──────────┘ └──────────┘ └──────────┘ └────┬─────┘         │
│                                               │ 到达触发        │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────▼─────┐         │
│  │  World   │ │   PvE    │ │  Hero    │ │ Combat   │         │
│  │ 地图/村庄 │ │ 目标/内容 │ │ 英雄     │ │ 战斗结算  │         │
│  │ (F组)    │ │ (E组)    │ │ (G组)    │ │ (D组后半) │         │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘         │
└───────────────────────────┬─────────────────────────────────┘
                            │ 登记/触发
┌───────────────────────────▼─────────────────────────────────┐
│  基础设施层 (Infrastructure) — 不含游戏逻辑                     │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐              │
│  │ Scheduler  │  │ EventBus   │  │ Persistence│              │
│  │ 时间调度器  │  │ 事件总线    │  │ 存储/惰性算 │              │
│  └────────────┘  └────────────┘  └────────────┘              │
└───────────────────────────────────────────────────────────────┘
```

### 各领域模块的状态归属（谁拥有什么数据）

| 模块 | 拥有的状态 | 不碰别人的 |
|------|-----------|-----------|
| **Economy** 经济 | 每村资源存量、产率参数、仓库容量、地窖 | 不管建筑等级（只读取产率结果） |
| **Building** 建筑 | 每村建筑槽位与等级、建造队列、科技解锁 | 不直接改资源（发命令请 Economy 扣） |
| **Military** 军队 | 每村兵力数量、训练队列、兵种科技等级 | 同上 |
| **Movement** 行军 | 在途部队（出发/到达/类型/载荷） | 结算时调用 Combat，不自己算战斗 |
| **Combat** 战斗 | 无持久状态（纯结算函数） | 输入双方快照，输出结果，不直接改库 |
| **World** 世界 | 地图格、村庄归属、坐标、绿洲/NPC点位置 | |
| **PvE** | PvE 目标定义、关卡/战役进度、刷新规则 | 战斗仍交给 Combat |
| **Hero** 英雄 | 英雄属性、装备、经验、状态 | |

> **关键**：Combat 是**无状态纯函数**——给它"进攻方快照+防守方快照"，它只返回"结果"（谁损失多少、抢多少、拆什么），由调用方(Movement)把结果分发给各 owner 模块去改状态。这样 PvE 和 PvP 能共用同一个 Combat（呼应 S0 的"同源"）。

### 2.1 派生属性管线（养成如何影响战斗——核心扩展模式）

> 这是回答"以后加养成系统怎么不破坏架构"的关键设计，借鉴所有 RPG/养成游戏的通用做法。

每个能影响战斗的模块（Hero、Military、未来的任何养成系统），内部都按**分层叠加**算出最终属性，对外只暴露一个**固定格式的"战斗贡献快照"**：

```
模块内部(以 Hero 为例):
   基础属性
 + 装备加成
 + 天赋树加成        ← 未来新增的养成系统，只在这里"加一层"
 + (任意未来养成再加一层)
 = 最终属性
       │
       ▼
   computeCombatContribution() ──▶ {攻击加成%, 防御加成%, ...} (固定格式快照)
       │
       ▼
   Combat 纯函数：只认快照，不知道"天赋树/装备"存在，更不知道怎么算的
```

**这带来的稳定性保证（加养成系统时的改动半径）：**

| 加一个新养成系统时 | 是否改动 |
|------------------|---------|
| 拥有它的模块(如 Hero)的属性计算 | ✅ 改（加一层加成，本就该改这里） |
| 该模块对外的"战斗贡献快照"格式 | ❌ 不改（数字变，格式不变） |
| Combat 模块 | ❌ 完全不动 |
| 其它所有模块 | ❌ 完全不动 |

**推广为全局原则**：

> **每个模块对外只暴露"算好的结果快照"，内部怎么算、叠了多少养成层，外部一概不知道也不关心。** 新功能的影响半径被锁在它该影响的模块内，绝不外溢——这就是"加十个养成系统，Combat 一行不改"。

这条原则同样适用于：Economy 对外只给"最终产率/存量"，不暴露怎么算的；Building 对外只给"最终建造耗时"，不暴露主基地/加成怎么叠的。

## 三、模块间通信：Command / Event 两种契约

> 通信的**信封格式**单独定义在 `04_通信格式规范.md`（已定稿）。本节只讲"为什么这么分"。
> 一句话：**固定外层信封，自由内层 payload**——升级=payload放资源，战斗=payload放资源+兵力，同一信封换内容。

### 通信规则
- **Command（命令，同步请求-响应）**：A 模块要 B 模块做事/给数据 → 发 Command，等返回。用于"必须立刻知道结果"的场景（如：建造前问 Economy"够不够资源、扣掉"）。
- **Event（事件，异步广播）**：某模块状态变了，广播出去，关心的模块自己订阅。用于"通知"（如：Movement 广播"部队已到达"，战报模块、客户端推送各自响应）。

### 为什么两种都要
- 纯事件：解耦但难处理"需要立即确认成功/失败"的操作（扣资源失败要当场拒绝）。
- 纯命令：直接调用会导致模块互相硬依赖。
- → **写操作用 Command（要确认），通知用 Event（要解耦）**。

### 一条端到端示例：玩家点"升级伐木场"

```
1. 客户端 → Gateway:  请求 UpgradeBuilding(村X, 伐木场)
2. Gateway → Building: Command: TryUpgrade(村X, 伐木场)
3. Building 校验前置/队列空位 → 算出花费
4. Building → Economy:  Command: TrySpend(村X, {木:.., 泥:..})
                        Economy 先惰性结算资源到当前，够则扣、返 OK；不够返 Fail
5a. 若 Fail → Building 拒绝 → Gateway 回客户端"资源不足"
5b. 若 OK  → Building 入队，→ Scheduler 登记事件 BuildDone(村X,伐木场, 触发时刻=now+T)
            → Gateway 回客户端"已开始建造，预计 XX 完成"
6. (时间到) Scheduler 触发 → Building 处理 BuildDone：等级+1
7. Building → EventBus 广播 Event: BuildingUpgraded(村X, 伐木场, 新等级)
8. Economy 订阅该事件 → 更新该村木产率；Gateway 订阅 → 推送客户端刷新
```

这条链把三条铁律全用上了：状态各归各家、跨模块只传 Command/Event、时间走 Scheduler。

## 四、基础设施层（无游戏逻辑，所有模块复用）

| 组件 | 职责 | 为什么独立出来 |
|------|------|--------------|
| **Scheduler 调度器** | 登记"未来某时刻触发某事件"，到点回调对应模块。崩溃后能从持久化恢复重建 | 全游戏唯一时间源，避免各模块自己藏定时器导致不可控/不可恢复 |
| **EventBus 事件总线** | 模块广播 Event，订阅者接收 | 解耦：发布者不需要知道谁在听 |
| **Persistence 存储** | 落库、读取；封装"惰性结算"读取模式（资源等连续量） | 把"读时计算"模式收口到一处，领域模块不重复实现 |

## 五、扩展点（为"以后好改"预留的位置）

这些是架构里**故意留的活口**，未来加内容不用改架构：

| 扩展点 | 怎么留 | 用来加什么 |
|--------|--------|-----------|
| **资源类型** | Economy 用"资源类型配置表"，不写死字段 | 2资源→4资源，加新资源 |
| **兵种** | Military 用"兵种定义表"(三维/速度/造价/科技) | 加兵种、加部族兵种 |
| **建筑** | Building 用"建筑定义表"(前置/产出/耗费) | 加建筑、改科技树 |
| **PvE 目标** | PvE 用"目标定义表"(怪/营地/关卡，含掉落规则) | 加 PvE 内容、战役 |
| **行军类型** | Movement 用"类型→结算策略"映射 | 加新行军类型(未来驻援、占领) |
| **战斗流水线** | Combat 用"可插拔步骤"(侦察→主战→拆墙→掠夺) | 增减战斗阶段、加战术层 |
| **部族** | 兵种/建筑表加"部族"维度 | 从1族扩到多族 |
| **后置系统接入** | 预留 Event 钩子(如战斗结果广播) | 联盟/赛季/商业化未来订阅即可接入，不改核心 |

> 原则：**内容用配置表，行为用可插拔策略，跨界用 Event 钩子**。三者保证扩展不动架构。

### 5.1 扩展性验证案例（用户预期要加的系统）

> 用三个真实预期功能验证架构扛不扛得住。结论：**都不用改 Combat**。照此模式加东西即可。

**① 英雄养成（装备/天赋/技能）—— 架构直接支持**
- 装备、天赋、技能(数值类)都是 Hero 内部的加成层，走派生属性管线。
- 改动半径：**仅 Hero 内部**。Combat 及其它模块零改动。

**② 部队养成（突破/专精/羁绊）—— 已留扩展点**
- 与英雄同构，owner 是 Military。原版"铁匠铺升攻防"已是一层，新增养成再叠层。
- 改动半径：**仅 Military 内部**。Combat 零改动。

**③ 工会 —— 新增模块 + 派生管线多认一个 buff 来源**
- 工会是**全新状态 owner**（成员/等级/工会建筑/工会 buff），需新增 Guild 模块。
- 工会 buff 不直接塞进别人模块，而是各模块派生管线里**多加一层"工会 buff 来源"**：Economy 算产率多问"工会给产量 buff 吗"，Hero/Military 算属性多加一层工会 buff。
- 工会通过订阅 Event（如"成员战斗胜利"）更新自身状态。
- 改动半径：新增 Guild 模块 + 各模块派生管线多认一个来源。**Combat 零改动**。

### 5.2 两种养成类型（设计每个养成时需明确属于哪种）

| 类型 | 例子 | 架构怎么接 | 改 Combat？ |
|------|------|-----------|------------|
| **静态数值型**（绝大多数） | 装备+攻击、天赋+10%防、工会+产量、部队专精 | 派生管线加一层 | ❌ 不用 |
| **战斗触发型**（少数） | 技能"开战眩晕"、"血量<30%狂暴" | Combat 流水线加一个可插拔步骤（流水线本就设计成可插拔） | ⚠️ 加插件步骤，不改主逻辑 |

> 设计任一养成时，先归类：静态数值型零成本；战斗触发型写成一个流水线步骤插件。两者都不触碰 Combat 主逻辑。
