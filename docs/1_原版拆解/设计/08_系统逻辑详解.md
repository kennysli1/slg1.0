# 08 · 系统逻辑详解（架构基线 / Mechanics Bible）

> 本文是策划案的"逻辑核心层"，把每个系统讲清楚到**可实现**的程度：实体、状态机、事件流、规则边界、系统间依赖。
> **数值一律用符号/占位**（如 `T_build`、`R_prod`），真实数值见 `05` 且可随意调整。本文只锁"逻辑骨架"，逻辑确定后数值才有意义。

---

## 0. 四条总设计原则（所有系统都遵守）

这是整个游戏架构的地基，**任何系统设计都不得违反**：

1. **服务器权威 (Server-Authoritative)**
   一切状态变更与结算都在服务端完成；客户端只发"意图(指令)"和"展示状态"。防作弊的根。

2. **时间即资源 (Time as Resource)**
   一切有价值的动作都需要真实时间完成：建造、训练、行军、研究。加速时间 = 核心策略/付费点。

3. **惰性结算 (Lazy Evaluation)**
   连续变化的状态（最典型是资源）**不每秒更新数据库**，而是记录"上次结算时刻 + 当时的值 + 速率"，在**被读取或被事件触发时**才按 `Δt` 一次性算出当前值。
   → 这是性能与一致性的命门，下文资源系统详述。

4. **离散事件驱动 (Discrete Event-Driven)**
   一切"未来会发生的事"（建造完成、行军到达、训练出兵、忠诚恢复）都登记为**带触发时间戳的事件**，由调度器到点触发结算。游戏世界 = 一个按时间推进的事件队列。

> 记住这组对偶：**连续量用惰性结算（读时计算），离散事件用调度器（到点触发）**。整个 Travian 的运行时就这两类东西。

## 2. 时间与事件系统（运行时核心）

### 2.1 两类时间状态

| 类型 | 例子 | 存储方式 | 读取方式 |
|------|------|----------|----------|
| **连续量** | 4 种资源存量、英雄经验自然增长 | 存 `(基准值, 基准时刻, 速率)` | 读时按 `Δt` 算 |
| **离散事件** | 建造完成、行军到达、训练出兵、忠诚恢复、庆典结束 | 存 `(类型, 触发时刻, 载荷)` 入事件表/队列 | 调度器到点触发 |

### 2.2 事件生命周期（状态机）

```
   创建(玩家下指令/系统生成)
        │  计算 trigger_time = now + duration
        ▼
   ┌─────────┐   到达 trigger_time   ┌───────────┐
   │ PENDING │ ───────────────────▶ │ TRIGGERED │
   └─────────┘                       └─────┬─────┘
        │ 取消(若允许)                       │ 执行结算逻辑
        ▼                                   ▼
   ┌──────────┐                       ┌──────────┐
   │CANCELLED │                       │ RESOLVED │ → 可能产生新事件
   └──────────┘                       └──────────┘
```

- **结算可能链式产生新事件**：例如"行军到达"结算后，若是攻击，立刻生成一条"返程"事件。
- **幂等性要求**：每个事件结算必须可安全重试一次（崩溃恢复时重放），通过 `processed` 标志位保证只生效一次。
- **结算顺序**：同一时刻的多个事件，按确定顺序处理（如 `trigger_time` 相同则按 `id`），保证可复现。

### 2.3 实现形态（逻辑层，不绑技术）
- 近期事件 → 内存定时器/时间轮，毫秒级精度。
- 远期事件 → 持久化事件表，由扫描器拉取临近事件入内存。
- 服务重启 → 从事件表恢复所有 PENDING 事件，按时间重建调度。

> **这一节是全游戏的"心脏"**：后面所有系统（资源/建造/行军/战斗）都只是"往事件系统里登记事件 + 在事件触发时改状态"。

## 3. 资源系统逻辑

### 3.1 状态存储（惰性结算的具体落地）

每村每种资源只存三个量 + 一个时间戳：

```
village_resource {
  amount_at_tick   // 上次结算时的存量
  last_tick        // 上次结算时刻
  // 速率不存死值，每次按当前建筑/田等级/加成实时求
}
```

### 3.2 读时结算算法

```
function getResource(村, 资源类型, now):
    rate     = 计算当前每秒产量(资源类型)      // 资源田等级 + 强化建筑% + 绿洲% + 英雄% + 服务器速率
    capacity = 计算当前容量(资源类型)          // 仓库/粮仓等级
    elapsed  = now - last_tick
    新值 = amount_at_tick + rate * elapsed
    return min(新值, capacity)                // 关键：溢出截断，超出部分浪费
```

- **写时刻**：任何"花资源/加资源"的动作（建造下单、训练、被掠夺、运送到达）发生时，先 `getResource` 结算到当前，再扣/加，再更新 `amount_at_tick` 和 `last_tick`。
- **溢出**：存量达容量上限后停止增长（浪费），是逼玩家"花资源/扩仓"的压力来源。

### 3.3 粮食(Crop)的特殊逻辑——净产出与饿死

Crop 与其他三资源最大不同：**它被持续消耗**。

```
crop净产率 = Σ农田产出 - Σ(每个建筑/资源田等级的人口消耗) - Σ(每个士兵的耗粮)

if crop净产率 >= 0:  crop 正常增长至上限
if crop净产率 <  0:  crop 持续下降；
                     当 crop 存量触底(=0) → 触发"饥荒"事件：
                        按规则定期饿死部分军队，直至净产率回正
```

- **设计意图**：crop 是军队规模的硬天花板，防止无限爆兵；也是攻击战术（投石机拆农田 → 敌方饿死军队）。
- **逻辑要点**：饿死是一个**周期性事件**（如每小时检查一次），不是连续的，符合事件驱动原则。

### 3.4 隐藏地窖 (Cranny)
- 每个地窖提供一定"隐藏容量"，被掠夺时这部分资源不可被抢走。
- 结算掠夺时：`可抢量 = max(0, 当前存量 - 地窖隐藏容量)`。
- 反制：条顿斥候/攻击对地窖隐藏量有削减系数（侦察方部族修正）。

## 4. 建筑系统逻辑

### 4.1 建造队列状态机

```
玩家点"升级建筑X"
   │ 校验：① 前置建筑达标？ ② 资源够？ ③ 队列有空位？
   │  ── 任一不满足 → 拒绝，返回原因
   ▼ 全满足 → 立即扣资源(锁定) + 入队
┌──────────┐  排到队首且开始   ┌──────────┐  到达finish_time  ┌──────────┐
│  QUEUED  │ ───────────────▶ │ BUILDING │ ────────────────▶│  DONE     │
└──────────┘                   └────┬─────┘ (登记完成事件)    └──────────┘
                                    │ 玩家取消(可选)            建筑等级+1
                                    ▼ 退还部分资源              人口/CP/产量更新
                               ┌──────────┐
                               │ CANCELLED│
                               └──────────┘
```

- **资源在下单时即扣除**（锁定），不是完成时扣 —— 防止"排队后花光资源赖账"。
- **建造时长** `T = T_base(类型,目标等级) / 主基地加速系数 / 服务器速率`。
- **完成事件触发时**：等级+1 → 重算该村人口、CP 产率、资源产率/容量；若有连锁解锁（新建筑可建）则刷新可建列表。
- **队列条数**：免费 N 条（原版 1），付费可加。队列满则不能再下单。

### 4.2 前置依赖（科技树）
- 每个建筑定义 `prerequisites: [(建筑类型, 最低等级), ...]`。
- 可建判定：玩家请求时实时校验前置。前置被投石机拆到低于要求时，已有建筑保留但不能再升（设计裁定，见开放问题）。

### 4.3 拆除/降级
- 主基地可主动拆除其他建筑（逐级降，按时间）。
- 投石机攻击可强制降级/摧毁指定建筑（见战斗系统）。

## 5. 兵种训练逻辑

```
玩家在 兵营/马厩/兵工厂 下单训练 N 个兵种X
   │ 校验：① 该兵种已研究(学院)？ ② 资源够 N 个？ ③ 建筑存在？
   ▼ 扣资源(一次性扣 N 份) → 入训练队列
┌──────────┐  逐个完成(每 T_train 出一个)  ┌──────────┐
│ TRAINING │ ─────────────────────────────▶│ 入村驻军 │
└──────────┘   每出一个 → 登记下一个完成事件  └──────────┘
```

- **逐个产出**：训练 N 个不是一次性，而是每隔 `T_train` 出 1 个并登记下一个事件；玩家可中途看到部分完成。
- **资源一次性预扣**（下单即扣 N 份），取消未完成部分可退。
- **crop 约束**：新兵加入即增加耗粮，可能使净产率转负 → 联动饿死逻辑。
- **科技**：学院研究解锁兵种，铁匠铺逐级提升某兵种攻/防（永久，影响所有该兵种战斗结算取值）。

## 6. 行军系统逻辑（最复杂、跨村交互的唯一通道）

### 6.1 行军类型 (Movement Types)

| 类型 | 目的 | 到达后行为 | 是否返程 |
|------|------|-----------|----------|
| 攻击 Attack | 正常进攻，清军/拆建筑 | 战斗结算 | 幸存者返程 |
| 掠夺 Raid | 抢资源，损失更小 | 战斗结算(低损) + 抢资源 | 幸存者带资源返程 |
| 驰援 Reinforce | 帮友军防守 | 并入目标村驻防 | 不返（除非召回） |
| 侦察 Scout | 刺探情报 | 侦察战 + 生成情报 | 返程 |
| 建村 Settle | 拓荒者建新村 | 在目标空地建村 | 不返（消耗拓荒者） |
| 占领 Conquer | 行政官降忠诚 | 战斗 + 降忠诚度 | 幸存者返程 |
| 运输 Transport | 商队送资源 | 资源入目标村 | 商队返程 |
| 召回/返程 Return | 系统生成 | 兵/资源回到源村 | — |

### 6.2 行军状态机

```
玩家发兵(选目标+类型+兵力[+资源])
   │ 校验：兵力足够？目标合法？(运输:资源够?)
   │ 计算 arrive_time = now + 距离 / 最慢单位速度 / 加成
   │ 从源村"扣出"参战兵力(标记为在途，不在村)
   ▼
┌──────────┐  到 arrive_time  ┌──────────┐  结算   ┌──────────┐
│ OUTBOUND │ ───────────────▶│ ARRIVED  │ ──────▶ │ RESOLVED │
└────┬─────┘                  └──────────┘         └────┬─────┘
     │ 可召回(仅未到达前,                                  │ 生成战报/情报
     │  返程时间=已飞行时间)                                │ 按类型生成返程事件
     ▼                                                    ▼
┌──────────┐                                        ┌──────────┐
│ RECALLED │                                        │ RETURNING│→到达后兵/资源入村
└──────────┘                                        └──────────┘
```

### 6.3 关键规则
- **速度取最慢**：整支队伍按最慢单位速度行进（带投石机就很慢）。
- **距离/时间**：欧氏或环面距离 `÷` 速度 `÷` 加成（锦标场远程加速、英雄坐骑、神器）。
- **波次(Wave)**：玩家可在极短间隔连发多条 Movement 实现"清防波 + 占领波"连击 —— 高玩核心技巧，也是"半夜操作"痛点来源（2.0 备选改进点）。
- **兵在途即离村**：发出后这些兵不在源村防御序列里，源村空虚 —— 进攻=承担本村被反打的风险。

> 行军系统是把"村庄"这些孤岛连成"博弈网络"的唯一机制。它 + 战斗系统 = PvP 的全部。

## 7. 战斗结算逻辑（行军到达时一次性结算）

战斗**不是回合对拼**，是行军到达瞬间按公式一次算完。结算严格按以下**顺序流水线**：

```
1. 侦察拦截
   若来犯含/为侦察 → 先算斥候对抗(攻方斥候 vs 守方斥候)
      攻方侦察赢 → 生成情报(看到资源/兵力/建筑)，可能不被发现
      守方斥候赢 → 拦截，攻方斥候有损失，守方收到"被侦察"报警

2. 集结防御方总兵力
   守方 = 本村驻军 + 所有驰援部队 + 英雄(若在) + 陷阱(高卢)
   分别汇总 对步防 / 对骑防

3. 计算攻防比 → 主力战斗结算
   攻方按兵种攻击力求和 × (士气 × 英雄 × 啤酒节等加成)
   守方按"攻方步/骑构成比例"加权取 对步防/对骑防 × (1+城墙%)
   按非线性公式得双方损失率(见 05) → 扣减双方兵力

4. 城墙结算
   攻方含攻城锤 → 按命中降低守方城墙等级(先于/同步主战影响防御)

5. 攻城结算(仅攻方胜且含投石机)
   投石机按指定目标摧毁/降级建筑(可指定:随机/特定/资源田)

6. 占领结算(仅攻方胜且含行政官)
   行政官按数量降低目标村忠诚度;忠诚度→0 则该村易主(归属变更)
   (首都村不可被占;王宫/官邸防内奸提供忠诚恢复与抗性)

7. 资源掠夺(攻方胜)
   可抢量 = max(0, 守方资源 - 地窖隐藏量)
   按攻方幸存兵力总载货量上限搬运;不足则部分搬

8. 生成战报 + 返程事件
   双方各得一份战报(信息详略取决于侦察结果)
   攻方幸存者(+掠夺资源) 生成 RETURNING 事件回源村
   驰援的幸存者留在守方村继续驻防
```

- **顺序很重要**：先侦察→再主战→再拆墙/拆楼→再占领→再抢资源。实现时按此流水线，每步基于上一步结果。
- **多波同时到达**：按 `arrive_time` 排序逐波结算，前一波的损失影响后一波（连击战术的逻辑基础）。
- **士气/人口保护**：大号打小号攻击力打折，护新手。

## 8. 扩张系统逻辑（村庄数量增长）

### 8.1 文化点 (CP) 门控

```
CP 是连续累积量(惰性结算):  每村建筑按等级产 CP/小时,玩家所有村累加
开第 n 个新村/占第 n 个村 需要 累计CP ≥ threshold(n)   // threshold 随 n 递增
庆典(市政厅)可一次性大量增加 CP → 加速扩张
```

→ CP 把"扩张速度"和"基建投入"绑定，防止无脑铺村。

### 8.2 两种扩张路径

**A. 建立新村 (Settle)**
```
条件: 累计CP达标 + 拥有王宫/官邸(可派拓荒) + 训练满3个拓荒者 + 目标格为空
流程: 发 Settle 行军 → 到达 → 在空格创建新村(基础资源田Lv0/主基地Lv1) → 拓荒者消耗
```

**B. 占领他村 (Conquer)**
```
条件: 累计CP达标 + 拥有行政官(元老/酋长/首领) + 反复攻击降目标忠诚度至0
流程: 多次"占领"行军 → 每次战斗胜+行政官把忠诚度往下压 → 归0瞬间易主
反制: 守方王宫/官邸提供忠诚度自然恢复(事件:周期性回升) + 抗性; 首都不可被占
```

### 8.3 忠诚度 (Loyalty) 状态
- 连续量 / 周期事件混合：被行政官压低后，随时间**周期性恢复**（登记恢复事件）。
- 归 0 即触发"易主"事件：village.player_id 变更，驻军/队列处理按规则（通常清空敌对队列）。

## 9. 英雄系统逻辑

```
Player 1:1 Hero
状态: 存活(在某村/随某军行军中) / 阵亡(可在英雄殿复活,需时间+资源)
经验来源: 战斗杀敌、冒险 → 升级 → 获得属性点自由分配
属性: 战斗力 / 攻防加成% / 资源产出 / 冒险经验 → 影响所属系统取值
装备: 武器/护甲/坐骑/消耗品(加速/扩载/复活卷) → 槽位制,可换,可在银币市场拍卖
```

- **冒险 (Adventure)**：地图周期性刷新冒险点(事件) → 英雄出征(行军) → 到达结算掉落(装备/资源/经验)，有风险(英雄受伤)。
- **绿洲占领**：英雄带兵清绿洲野怪(战斗) → 胜则可在英雄殿绑定绿洲 → 给村庄加资源产率%（联动资源系统）。
- 英雄随军时计入战斗结算的加成项（见战斗第2/3步）。

## 10. 联盟与外交逻辑

Alliance: 成员上限 M(原版60); 超出→拆分wing组成meta联盟

## 11. 终局与赛季逻辑

```
赛季时间线(事件驱动的世界级里程碑):

 注册期──新手保护──自由发展──┬─[~Day100] 神器在Natars村刷新(世界事件)
                              │     玩家/联盟攻破Natars夺神器 → 获全局/单村加成
                              │
                              ├─[~Day200] 世界奇迹图纸在Natars刷新
                              │     夺图纸 → 在特定村可建"世界奇迹"
                              │     各阵营疯狂建造(升级事件) + 互相投石机拆奇迹
                              │
                              └─[某玩家奇迹达Lv100] → 触发"赛季结束"事件
                                    全服冻结(禁建造/交易/战斗) → 结算名次
                                    → 归档数据 → 开新世界实例(shard)
```

- **神器/奇迹/Natars** 都是**系统生成的世界级事件**，由赛季调度器按天数触发，不依赖玩家。
- **跨赛季继承**（外观/成就/英雄皮肤等，不继承战力）是 2.0 的开放设计点（见 `06` B7）。
- **赛季重置 = 新建一个 World 实例**，旧实例归档只读。这与"分服(shard)"架构天然契合。

## 12. 系统间依赖总图（谁影响谁）

```
            ┌─────────────────────────────────────────┐
            │           时间/事件系统 (§2)              │
            │   所有系统都向它登记事件、被它触发结算      │
            └───────────────────┬─────────────────────┘
                                │ 驱动
   ┌──────────┐  产/耗   ┌──────▼──────┐  消耗   ┌──────────┐
   │ 资源系统  │◀────────│  建筑系统    │────────▶│ 训练系统  │
   │  (§3)    │────────▶│   (§4)       │         │  (§5)    │
   └────┬─────┘  容量/CP └──────┬──────┘          └────┬─────┘
        │ 限制爆兵(crop)         │ 解锁/加成              │ 产出兵
        │                       │                       ▼
        │                  ┌────▼─────┐           ┌──────────┐
        │                  │ 扩张系统  │           │ 行军系统  │
        │                  │  (§8)    │◀──────────│  (§6)    │
        │                  │ CP/忠诚  │  占领改归属 └────┬─────┘
        │                  └──────────┘                │ 到达触发
        │ 掠夺减资源                                     ▼
        │                                         ┌──────────┐
        └─────────────────────────────────────────│ 战斗系统  │
                            英雄(§9)随军加成 ──────▶│  (§7)    │
                            联盟(§10)驰援汇总 ─────▶└────┬─────┘
                                                        │ 胜负/占领/掠夺
                                                        ▼
                                                  ┌──────────┐
                                                  │ 终局(§11) │
                                                  └──────────┘
```

**一句话总结架构**：
> **事件系统是心脏**；资源/建筑/训练构成"经济引擎"，受 crop 与 CP 两道闸门约束；行军是唯一的跨村通道，把孤岛连成博弈网；战斗在行军到达时按固定流水线结算，其结果回流到资源(掠夺)、村庄归属(占领)、终局(奇迹)。所有系统的"变化"都不是连续 tick，而是"惰性结算 + 到点事件"。

## 13. 本文锁定的待裁定逻辑点（影响实现，需你拍板）

> 这些是**逻辑层面**的开放问题（区别于 `06` 的玩法方向问题），需要明确规则：

1. **前置被拆**：建筑前置被投石机拆到不达标时，依赖它的高级建筑是保留还是连锁降级？（建议：保留但锁升级）
2. **在途召回**：行军可召回的时间窗是"任意未到达前"还是"发出后X秒内"？返程时间如何算？
3. **队列取消退款比例**：建造/训练取消退还多少资源？（建议：未开始全退，进行中部分退）
4. **饿死规则**：crop 触底后按什么顺序、什么比例饿死哪些兵？（影响防守心理）
5. **占领后处理**：占领他村后，原主的队列/驰援/在途军队如何处置？
6. **同刻事件定序**：完全同一 `trigger_time` 的事件用什么二级键定序保证可复现？（建议：事件id）

→ 这些可以等基础策划案整体成型后统一裁定；我先按建议默认值往下设计。
