# 07 · 开发计划（Travian 2.0）

> 本计划随 `06` 的决策更新。当前为**基线版**（假设 2.0 保留 Travian 核心骨架 + 现代化技术栈 + 可调改进）。

## 1. 技术架构（建议基线）

Travian 本质是**服务器权威的实时模拟 + 定时事件队列**。关键约束：成千上万玩家的建造/行军/训练都是"在未来某时刻触发"的事件，且离线也要推进。

```
┌─────────────┐     WebSocket/HTTP      ┌──────────────────────┐
│  客户端      │ <───────────────────>   │  API 网关 (Node/Go)   │
│ Web + 移动端 │                          └──────────┬───────────┘
│ (React/TS)  │                                     │
└─────────────┘                          ┌──────────▼───────────┐
                                          │  游戏逻辑服务         │
                                          │  - 资源结算(惰性计算) │
                                          │  - 战斗结算           │
                                          │  - 建造/训练/行军     │
                                          └──────┬───────┬───────┘
                                                 │       │
                              ┌──────────────────▼──┐ ┌──▼──────────────┐
                              │ 定时事件调度器        │ │ 持久化           │
                              │ (job queue / 时间轮) │ │ PostgreSQL +     │
                              │ 行军到达/建造完成     │ │ Redis(热数据/锁) │
                              └──────────────────────┘ └─────────────────┘
```

**关键技术决策**：
- **惰性资源结算**：资源不每秒写库，而在玩家访问/事件触发时按"上次结算时间 → 现在"批量计算。性能核心。
- **事件驱动**：建造完成、行军到达、训练完成均为定时任务（延迟队列/时间轮）。
- **服务器权威**：所有结算后端算，前端只展示，防作弊。
- **分服(Shard)**：每个赛季 = 一个独立世界实例，便于水平扩展与赛季重置。

**技术栈建议（可换）**：
| 层 | 建议 | 备选 |
|----|------|------|
| 前端 | React + TypeScript + Vite | Vue / 原生App(RN/Flutter) |
| 后端 | Node.js(NestJS) 或 Go | Java(Spring) |
| 实时 | WebSocket (Socket.io / native) | SSE 轮询 |
| 数据库 | PostgreSQL | MySQL |
| 缓存/锁/队列 | Redis | RabbitMQ + Redis |
| 部署 | Docker + K8s，按赛季分实例 | 单机起步 |

## 3. 里程碑路线图

### M0 · 设计冻结（依赖 `06` 决策）
- 输出：确定 2.0 范围的最终 GDD（`01`~`05` 的 2.0 版）。
- 关键产物：兵种/建筑/数值终表、系统取舍清单。

### M1 · 经济核心可玩（单村 Demo）
- 一座村庄、4 资源、资源田升级、惰性资源结算、仓库/粮仓上限、建造队列。
- 前端：村庄视图 + 资源视图 + 建造交互。
- **里程碑标准**：能登录 → 升级资源田 → 资源随时间增长 → 建造完成触发。

### M2 · 军事与地图
- 兵种训练、地图(x,y)与村庄分布、行军系统(发兵/到达事件)、侦察。
- 战斗结算引擎 + 战报。
- **里程碑标准**：A 村可派兵掠夺 B 村，结算并生成双方战报。

### M3 · 扩张与社交
- 文化点、建立新村、占领村庄、市场交易、英雄与冒险、联盟与援助。
- **里程碑标准**：玩家可开二村、可加联盟、可互相驰援防守。

### M4 · 终局与赛季
- Natars、神器、世界奇迹、赛季胜利判定与重置流程、跨赛季继承(若 B7 选用)。
- **里程碑标准**：一个压缩时长的内测赛季能完整跑到"奇迹满级=结束"。

### M5 · 商业化与打磨
- 付费(按 B6 决策)、战令/段位(若选用)、QoL、移动端适配、反作弊、运营后台。

### M6 · 封测 → 公测
- 平衡校准（用 `05` 旋钮）、压测、监控告警、客服/举报工具。

## 4. 排期（粗估，按团队规模缩放）

> 假设小团队（2~4 工程 + 1 策划 + 1 美术），里程碑串行 + 部分并行。

| M3 扩张/社交 | 5~8 周 | 联盟/占领的边界 case |
| M4 终局/赛季 | 4~6 周 | 赛季重置数据流程 |
| M5 商业化/打磨 | 4~6 周 | 付费平衡 |
| M6 封测→公测 | 持续 | 平衡与稳定性 |

## 5. 横切关注点（贯穿全程）
- **反作弊/反脚本**：服务器权威 + 行为检测（农场脚本是 Travian 顽疾）。
- **平衡可调**：所有数值走配置表（对应 `05` 旋钮），支持热调。
- **可观测性**：日志、指标、赛季健康度看板（活跃、战力分布、通胀）。
- **数据安全与合规**：账号、支付、未成年保护。
